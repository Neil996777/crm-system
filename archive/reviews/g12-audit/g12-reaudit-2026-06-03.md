## Security-group verification (BLK-G12-003 / 007 / BLK-G11-002)

The 1st audit found Codex's "dedicated-raw" file (20:29) recorded only `Describe` calls — no
`CreateSecurityGroup`/`AuthorizeSecurityGroupIngress`/`RevokeSecurityGroupIngress`/
`ModifyNetworkInterfaceAttributes` RequestIds — so it could not prove a mutation. Root cause:
the operator account `crm-infra-readonly` (the project's single sub-account) initially lacked
`vpc:*` write actions (genuine `AccessDenied` at 20:14). After the user granted full
permissions, Claude ran an **independent read-only live Volcengine query** (Describe only, no
mutation) and captured the raw API responses:

- ENI `eni-13e8tbocd8f0g79iu5jer8idt`.SecurityGroupIds = `['sg-366ptx1bxp9ts1e710babmc8y']`
- Dedicated SG `sg-366ptx...` (`crm-system-prod-public`) public TCP ingress = **22, 80, 443 only**
- No public `8080`/`5432`; Default SG no longer bound to the CRM ENI
- Real RequestIds, `2026-06-03T22:08+08:00`, http=200
- Evidence: `docs/release/evidence/volcengine-security-group-verified-readonly-2026-06-03.json`

The **underlying remediation is real and now independently verified**; it is accepted on the
strength of this Claude-run live read. The *provenance* weakness (no captured mutating calls)
and the *checker* weakness are carried as BLK-G12-009 — they do not reopen the network state,
which is confirmed correct.
