## 1. What / Why
"列表页"是 CRM 的核心页型之一。本轮以**商机（Opportunity）列表**作为该页型的范例，确立列表页的
信息密度、筛选/检索、批量操作、分页、空/载入态等通用范式，供后续其他实体列表复用。
用户：销售（看自己的商机）/ 销售经理（看团队）/ 管理员（看全部）。
目的：在一屏内高效浏览、筛选、批量处置商机，并能下钻到单条。

## 2. 必须呈现的内容（功能要求，非设计指令）
页面须让上述用户能完成以下，具体形态由 Codex 设计：
- **页面骨架**：复用已锁定的 v7 工作台外壳（展开侧栏，"商机"为 active；64px 顶栏；zh-CN）。
- **页头**：标题"商机" + 结果计数 + 主操作（新建商机）+ 次操作（导出）。
- **检索与筛选**：关键词搜索 + 按 阶段 / 负责人 / 金额 / 日期 / 状态 等维度筛选；须能表达
  "已应用筛选"与"清除筛选"。
- **列表主体**：逐条商机，含足以支撑销售判断的关键字段（至少：商机名、客户、阶段、负责人、
  金额、预计签约、更新时间）。金额须 tabular 对齐 + ¥ 前缀。
- **选择 + 批量操作**：可多选；选中后出现批量操作（如改负责人/改阶段/导出所选/归档）。
- **分页**：经典分页（非无限滚动）。
- **角色数据范围**（来自 interaction-spec Part B8）：销售=本人、经理=团队、管理员=全部。
- **默认排序**：更新时间倒序（UX 假设，可在设计中复核）。

## 3. 验收标准 / Definition of Done（= 我 G8 审计的标尺）
- **真实枚举（硬性）**：阶段列的值只能取商机六阶段枚举 —— 新商机 / 需求已确认 / 报价 /
  合同谈判 / 赢单 / 丢单（源：`frontend/src/i18n/labels.ts` `opportunityStageLabel`）。
  **不得**把回款/合同状态（如 逾期回款/部分回款/待签署/丢单风险）当作阶段值。
- **设计系统一致**：沿用已锁定 v4 设计系统，**不得新增/改动颜色或 token**；尽量复用既有
  组件类与表格样式（与三张工作台页视觉一致）。
- **状态覆盖**：至少呈现 默认行 / 选中行 / 悬停行 三态可辨；空态(empty)与载入态(loading)
  至少在源码注释中给出定义（是否单独出图见 §5 由 Codex 决定）。
- **真实感样本数据**：客户/负责人与既有 mock 宇宙一致；进行中商机的"预计签约"为未来日期；
  无明显自相矛盾的数据。
- **批量操作的安全语义**：不得提供"静默批量推进阶段"等危险无确认操作；终态/破坏性操作
  （归档等）须标注走确认流程（静态帧可不渲染确认弹窗，但须在注释标明）。
- **批量操作按角色门控（硬性，源：`services/opportunity/internal/domain/opportunity.go`）**：
  - 转移负责人：`CanEditOpportunity` 仅 经理/管理员可改 ownerID；销售被后端强制
    `newOwnerID == actorID`（不能转给他人）。故「批量转移负责人」**只能出现在经理/管理员
    视图**，销售视图必须隐藏。
  - 归档：`CanArchiveOpportunity` 仅 经理/管理员。「批量归档」同样**仅经理/管理员视图**显示。
  - 销售视图的批量操作集合应收敛为其权限内的安全动作（如 导出所选）。
  - 注：放开"销售可转移负责人（带新人场景）"属正式 Scope Change（动授权 + ACC + G12 安全
    不变量），须走 PM/安全评审，不得在设计层加入。
- **本地化**：全中文；不得回退英文，不得改动枚举/角色比较值。
- **PREVIEW-ONLY 合规**：单个自包含 HTML（仅内联 style + 内联 SVG）+ 其 PNG；未触碰
  `frontend/`、`src/`、构建配置、依赖。

## 4. 绑定约束
- No-downgrade：不弱化/删除任何 P0/P1 或既往 G12 修复。
- 锁定 v4 调色板，不得 recolor；"科技感"靠组件/数据呈现/交互，不靠改色。
- 保留 zh-CN（phase 1+2）。
- 本轮**只要 mock 图**，不做可点击原型。
- 该页型范例须可被后续其他列表复用（保持范式通用，不要做成商机特例）。

## 5. 明确移交给 Codex 决定的设计项
以下不在本规格约束内，由 Codex 设计判断（我只在 G8 审是否合理、是否满足 §3）：
- 列的精确集合/顺序/宽度、是否表格 vs 卡片行（建议表格，因多列需对齐，但由你定）。
- 阶段→badge 变体的具体映射（只能用现有 badge 变体，不得新增色）。
- 工具栏/筛选 chip 的布局与交互呈现、排序控件形态。
- 选中/悬停/空/载入 各态的视觉处理，以及是否把 empty/loading 单独出 PNG。
- 样本行数与具体样本内容（须满足 §3 的真实枚举与一致性）。
- 与 interaction-spec Part B8 一致的微交互（静态图中以注释体现即可）。

## 6. Out of scope
- 单条商机详情/表单页（后续页型）。
- 任何后端/API/数据模型/业务逻辑改动。
- 部署。

## 7. G8 审计我会核什么
对照 §3 逐条：真实六阶段枚举、无 recolor/新 token、状态覆盖、样本数据自洽、批量操作安全语义、
zh-CN、PREVIEW-ONLY 合规、以及与三张工作台页的视觉一致性与范式通用性。
